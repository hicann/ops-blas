# 硬规则

以下规则在任何阶段都必须遵守，不可跳过、不可变通。

1. **禁止主 Agent 亲自动手** - 每个步骤启动后，主 Agent 只负责调用对应的 Subagent：编码→`developer`，测试→`tester`，设计→`architect`，检视→`reviewer`，文档→`writer`。**禁止**在主 Agent 层面做以下操作：读模板文件（assets/*.md）、读代码、搜索仓内文件、抓取/搜索网页资料、写代码、写文档。这些全部是 Subagent 的工作。主 Agent 唯一例外：读取 references/ 下的流程参考文件、更新 LOG.md、发送 AskUserQuestion
2. **调用只定义三要素** — 输入、输出、验收标准，禁止干涉 Subagent 的实现细节
3. **阶段不可跳过** — 设计→开发→验收→上库，每阶段需用户确认后推进
4. **门控不可绕过** — Subagent 报告未通过验证时，禁止进入下一阶段
5. **问卷不允许修改** — 基于已生成的 json 模板发送 `AskUserQuestion` 问卷时，不得进行任何修改，直接发送
6. **问卷结果落盘** — 每次问卷得到用户答复后，保存为 `{原问卷名}.ret.json`，仅保留用户选择的选项，删除未选选项
7. **后期默认批准 + 失败直接打回** — CP1.2 之后的 1.4 / 2.1 / 2.2：（a）评审/验收通过 → 直接进入下一步，不询问用户；（b）评审/验收不通过 → 直接打回修订，不询问用户，同一轮超过 3 次仍不通过再生成 CP 问卷询问用户。CP3.2 和 CP4.3 除外，必须询问用户
8. **日志即时更新** — 每次流程步骤推进后，主 Agent 必须立即更新 LOG.md（当前步骤、进度状态、开发记录），不得延迟或累积到后续步骤
9. **修复只走 developer** — 若检视/验收报告发现问题或需要修复的建议，必须通过 developer Subagent 修复，主 Agent 不得直接修改代码
10. **CP 否定分支统一打回上一步** — 用户在 CP 问卷中选择「需要修改」时，按以下路径打回：

| CP | 「需要修改」打回 | 「终止流程」 |
|----|----------------|------------|
| CP1.1.A | 重新发送问卷 | — |
| CP1.1.B | 重新发送问卷 | — |
| CP1.2 | 打回 1.2 需求分析 | — |
| CP1.4 | 打回 1.3.A/1.3.B 重新设计 | 终止开发流程 |
| CP2.1 | 打回 2.1.1.A/2.1.1.B 重新开发 | 终止开发流程 |
| CP2.2 | 打回 2.2.1.A/2.2.1.B 重新开发 | 终止开发流程 |
| CP3.2 | 打回 3.1/3.2 重新检视/验收 | — |
| CP4.3 | 打回 4.2 代码检视 | — |

# 启动指令

**加载本技能后，主 Agent 必须立即执行以下动作，不得做任何其他操作（包括读文件、搜索、询问用户）：**

1. 确定当前步骤（首次加载从 1.1 开始，中断恢复从 LOG.md 记录的步骤继续）
2. 查找下方流程表中该步骤对应的"参与角色"
3. 调用该 Subagent，传入 task-prompts.md 中定义的输入/输出/验收标准
4. Subagent 返回后，主 Agent 更新 LOG.md，然后按流程表推进到下一步

# 流程全景

## 角色

| 角色 | 职责 |
|------|------|
| 用户 | 需求提出、各确认点审批 |
| writer | 资料准备、文档与问卷整理、文档编写 |
| architect | 需求分析、方案设计、方案评审 |
| developer | 代码开发、编译联调、性能调优 |
| tester | 测试设计、用例开发、测试验收 |
| reviewer | 代码检视：规范、一致性、风险 |

## 流程

| 步骤 | 输入 | 参与角色 | 输出 | 说明 | 并行 |

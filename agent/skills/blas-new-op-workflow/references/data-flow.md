# 数据流说明

## 文档存储

所有开发中间文档存放在 `.agent/dev-docs/{operator_name}/`：

```
.agent/dev-docs/{operator_name}/
├── LOG.md                      # 开发日志
├── 2.0.1-开发环境.md
├── 1.1-参考资料清单.md
├── 1.2-需求分析.md
├── 1.3.A-开发方案设计.md
├── 1.3.B-测试方案设计.md
├── 1.4.A-开发方案评审.md
├── 1.4.B-测试方案评审.md
├── 2.1.2-汇合联调报告.md
├── 2.1.3-测试验收报告.md
├── 2.2.2-汇合联调报告.md
├── 2.2.3-测试验收报告.md
├── 3.1-代码检视报告.md
├── 3.2-性能报告.md
├── 4.2-代码检视报告.md
├── 4.3-Issue.md
├── 4.3-上库PR模板.md
├── CP1.1.A.json / CP1.1.B.json / CP1.2.json ~ CP4.3.json
└── issues/
    └── issue_{YYYYMMDD}_{关键词}.md
```

## 外部参考数据源

```
.agent/cann-samples/Samples/    # cann-samples 仓库（只读，不产生中间文档）
.agent/asc-devkit/              # asc-devkit 仓库（只读，不产生中间文档）
```

通过 `op-samples-reference` 技能按需检索，在以下阶段作为参考输入：
- 阶段一（1.3.A 开发方案设计）：架构模式、优化策略选型
- 阶段二（2.1.1.A / 2.2.1.A 算子开发）：同类算子实现、编程模式
- 阶段三（3.2 性能验收）：性能调优实践、瓶颈分析方法

通过 `asc-devkit-reference` 技能按需检索，在以下阶段作为参考输入：
- 阶段一（1.3.A 开发方案设计）：API 文档查阅、示例代码参考、Tiling 策略设计
- 阶段二（2.1.1.A / 2.2.1.A 算子开发）：API 用法确认、同类算子实现参考、编程模式
- 阶段三（3.2 性能验收）：性能优化实践、瓶颈分析方法

---

## 阶段一：设计

```
用户需求
    │
    └─→ [1.1.A 资料准备] → 工作区目录(临时名) + LOG.md + 1.1-参考资料清单.md
            │
            └─→ [1.1.S 总结] → CP1.1.A.json
                    │
                    └─→ ⛔ CP1.1.A（算子名/数据类型/目标芯片）

CP1.1.A 确认算子名
    │
    └─→ [1.1.B 环境准备] → 2.0.1-开发环境.md + git 分支（{aclblasXxx}，使用 API 名）

CP1.1.A结论 + 1.1-参考资料清单.md
    │
    └─→ [1.1.S2 总结] → CP1.1.B.json
            │
            └─→ ⛔ CP1.1.B（精度标准/编程模型）

CP1.1结论 + 1.1-参考资料清单.md
    │
    └─→ [1.2 需求分析] → 1.2-需求分析.md + CP1.2.json
            │
            └─→ ⛔ CP1.2（用户审批：精度标准、约束确认）

1.2-需求分析.md
    ├─→ [1.3.A ⇄ 1.4.A 循环 ≤3] → 1.3.A + 1.4.A（开发侧：设计⇄评审）
    │
    └─→ [1.3.B ⇄ 1.4.B 循环 ≤3] → 1.3.B + 1.4.B（测试侧：设计⇄评审）

1.4.A + 1.4.B 双方通过
    └─→ ⚪ CP1.4（任一方3次失败→CP问卷）
```

## 阶段二：开发

```
1.3.A-开发方案设计.md → [2.1.1.A 算子开发] → 算子代码    ┐
1.3.B-测试方案设计.md → [2.1.1.B 测试开发] → 用例代码    ┤ 并行
                                                          │
算子代码 + 用例代码                                        │
    └─→ [2.1.2 汇合联调] → [2.1.3 测试验收] → ⚪ CP2.1    │
                                                          │
（迭代二结构相同）                                         │
```

## 阶段三：验收

```
git diff + OAT checklist + OAT 扫描报告（oat_reports/result.txt）+ 全部变更文件
    └─→ [3.1 代码检视] → 3.1-代码检视报告.md
            │
            ├─→ 通过：进入 3.2
            └─→ 不通过：developer 修复 → 回到 3.1 重新检视（≤3 次）
            │
            └─→ [3.2 性能验收] → 3.2-性能报告.md
                    │
                    └─→ ⛔ CP3.2（3.1 + 3.2 报告）
                        └─→ [3.3 大 shape 精简]（仅当用户选择「精简为 1 条」时执行）
```

## 阶段四：上库

```
git diff + OAT checklist + OAT 扫描报告（oat_reports/result.txt）+ 全部变更文件 + 文档
    └─→ [4.1 编写文档] → README.md
    └─→ [4.2 代码检视] → 4.2-代码检视报告.md
    └─→ [4.3 开发总结] → CP4.3.json + 4.3-Issue.md（内容来自 1.2-需求分析.md）+ 4.3-上库PR模板.md + 更新 LOG.md
            └─→ ⛔ CP4.3（AskUserQuestion → squash commit）
```

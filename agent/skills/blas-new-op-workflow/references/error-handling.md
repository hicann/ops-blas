# 错误处理指南

## 核心原则

1. **问题回退不跳过** - 验证失败时，修复后从当前阶段重新验证
2. **先定位后修复** - 不要盲目修改代码，先定位根因
3. **分层排查** - 按 Host → Kernel → 编译环境顺序排查
4. **记录问题** - 复杂问题创建 issue 文件，LOG.md 只放链接

## 重试阈值

| 验证类型 | 重试次数 | 说明 |
|---------|---------|------|
| 编译失败 | 3 次 | 语法或环境问题 |
| ST 精度失败 | 3 轮 | 每轮修复后重新验证 |
| 性能不达标 | 5 轮 | 优化迭代需要多次尝试 |
| 方案评审不通过 | 3 轮 | 打回设计修订后重评 |

## 阶段一：设计

### 1.2 需求分析

| 错误类型 | 处理方式 |
|---------|---------|
| 需求不完整 | 列出缺失项，请用户补充 |
| 接口信息不一致 | 查阅 include/cann_ops_blas.h 确认 |

### 1.3/1.4 方案设计与评审

开发方案（1.3.A ⇄ 1.4.A）与测试方案（1.3.B ⇄ 1.4.B）为两个独立并行循环，各自设计→评审→打回修订，不互相等待：

| 错误类型 | 处理方式 |
|---------|---------|
| API 不可用 | 查 ascendc-docs-search 找替代 API |
| UB 预算超限 | 减小 Buffer 或调整 Tiling 策略 |
| 评审不通过 | 直接打回对应1.3修订（1.4.A失败→打回1.3.A，1.4.B失败→打回1.3.B），不询问用户 |

### CP 问卷触发条件

⚪ 裁定步骤（1.4 / 2.1 / 2.2 / 3.2）中，任一路径/步骤连续 3 次不通过时：
- 第 1~2 次失败：直接打回修订
- 第 3 次仍失败：生成 CP 问卷询问用户

## 阶段二：开发

### 2.0.1 环境准备

| 错误类型 | 处理方式 |
|---------|---------|
| 缺少依赖（GCC/CMake 等） | 通过 AskUserQuestion 询问用户如何解决 |
| NPU 不可用 | 通过 AskUserQuestion 询问用户如何处理 |
| 分支/目录创建失败 | 检查权限，报告错误 |

### 编译错误

按以下顺序排查：
1. 检查代码语法错误
2. 检查头文件 include 路径
3. 对比仓内类似算子实现
4. 检查 CANN 环境变量 `ASCEND_HOME_PATH`

### ST 精度失败

按以下顺序排查：
1. 检查 golden 数据生成是否正确
2. 检查 TilingData 传递是否正确
3. 检查 DataCopy 参数（srcStride/dstStride/blockLen）
4. 启动 ascendc-precision-debug skill
5. 添加 Print 打印中间值定位

### 卡死/崩溃

启动 ascendc-crash-debug skill，检查：
1. Buffer 是否越界
2. TPipe EnQue/DeQue 配对
3. SetFlag/WaitFlag 事件 ID 匹配

## 阶段三：验收

### 代码检视不通过

直接打回修复，不询问用户。连续 3 次不通过 → 生成 CP 问卷。修复后必须重新执行 2.2.3 测试验收验证，确保修改未引入回归。

### 性能不达标

直接打回优化，不询问用户。连续 3 次不通过 → 生成 CP 问卷。启动 ops-profiling skill 排查：
1. 检查 MTE2/V/MTE3 利用率
2. 检查 Double Buffer 是否生效
3. 检查 bank conflict
4. 参考 ascendc-performance-best-practices skill 针对算子类型做专项优化

## 阶段四：上库

### 代码检视不通过

修复问题后需重新执行 4.2 代码检视验证。
